# コンテキスト最小化スキル

このスキルが有効な間は、以下のルールに従ってコンテキスト（トークン）消費を最小限に抑えて作業する。

## 基本原則

- **必要な情報だけを読み込む**。「念のため」でファイル全体やディレクトリ全体を読まない。
- **一度取得した情報は再取得しない**。会話内で既に確認済みの内容を再Read・再検索しない。
- **出力は要点のみ**。長い引用・全文貼り付け・冗長な経過報告をしない。

## ファイル読み取りのルール

1. **Read の前に必ず検索で位置を特定する**
   - `Grep` でシンボル・キーワードの行番号を特定してから、`Read` の `offset` / `limit` で該当部分（±30行程度）だけを読む。
   - ファイル全体の Read は、設定ファイルなど明らかに小さいファイル（目安: 100行未満）に限る。
2. **構造把握には Glob と Grep を使う**
   - ディレクトリ構成の把握は `Glob`(例: `app/**/*.tsx`) で済ませ、片っ端からファイルを開かない。
   - 「どこで使われているか」は `Grep` の `files_with_matches` モードで済ませ、内容が必要な場合のみ `content` モード + `head_limit` を使う。
3. **同じファイルを編集確認のために再読しない**
   - Edit / Write が成功していれば結果は保証されている。

## コマンド実行のルール

- 出力が大きくなるコマンドは必ず絞る: `| head -50`、`--oneline`、`--stat`、`-name` 指定など。
- `git log` は `--oneline -10`、`git diff` は `--stat` を先に見て、必要なファイルだけ個別に diff を取る。
- ビルド・テストのログは失敗箇所のみ抽出する（例: `2>&1 | grep -A 5 -i "error\|fail" | head -40`）。
- `cat` でのファイル全文表示は禁止（Read の部分読みを使う）。

## 探索のルール

- 広範囲の調査（複数ディレクトリの横断検索、命名規則の網羅確認など）が必要な場合は、`Explore` サブエージェントに委譲して結論だけを受け取る。自分のコンテキストに生の検索結果を溜めない。
- 検索は具体的なパターンから始める。ヒットが多すぎるパターン（例: `function`）で全件表示しない。

## 応答のルール

- 経過報告は1〜2文。ツール呼び出しの間に長い説明を書かない。
- 最終報告は「結果 → 変更点 → 補足」の順で簡潔に。コード全文の再掲はせず、`ファイルパス:行番号` で参照する。
- ユーザーに聞かれていない代替案・背景説明・次のステップ提案を長々と書かない。

## 禁止事項

- ファイル全体の無差別 Read（大きいファイルの limit なし Read）
- 同一内容の重複検索・重複 Read
- 大量出力コマンドの無加工実行（`git log` 全件、`find /` など）
- 作業に不要なファイルの「参考のため」の読み込み
